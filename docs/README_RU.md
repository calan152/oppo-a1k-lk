# LK для OPPO A1K (CPH1923)

Это рабочий порт MediaTek Little Kernel для **OPPO A1K / CPH1923**, проект
OPPO **18540**. Он получен не простой заменой конфигурации: поведение исходников
сверялось со стоковым `lk.bin`, UART, содержимым `expdb`, итоговым FDT и
регистрами дисплейного тракта.

Это смешанный по лицензиям исследовательский репозиторий. Корневой
[`LICENSE.md`](../LICENSE.md) описывает происхождение компонентов, но не
распространяет одну лицензию автоматически на каждый файл. Перед форком,
сборкой или публикацией прочитайте
[THIRD_PARTY_NOTICES.md](../THIRD_PARTY_NOTICES.md) и
[аудит распространения](REDISTRIBUTION_AUDIT.md).

## Что работает

- normal boot Android;
- recovery и fastboot;
- расширенное меню Normal/Recovery/Fastboot/META/Factory/Advanced META/ATE;
- панель `ili9881c_hd_dsi_vdo_txd_boe_zal1890`, подсветка и логотип;
- сохранение раннего LK, PL и ATF лога в 20-МиБ `expdb`;
- настоящий serial number из barcode;
- ранняя загрузка DTB из boot/recovery и наложение `dtbo`;
- RAM-загрузка normal boot и recovery через `fastboot boot`;
- чтение `expdb`, `boot`, `recovery`, `dtbo`, `system` и `vendor` через
  fastboot `fetch`;
- полноэкранное fastboot-меню с прогрессом операций;
- диагностические `getvar` для сборки, режима, WDT, панели, батареи и LK/LK2;
- полная проверка критической прошивки чтением назад и SHA-256;
- строгая проверка RAM boot-образов и тесты поврежденных заголовков на ПК;
- аппаратные кнопки CPH1923, назначение которых не зависит от DTBO;
- fail-closed защита от случайной блокировки без автоматической записи в
  `seccfg`;
- цветное пульсирующее подтверждение `fastboot flashing unlock` с безопасным
  выбором `CANCEL` по умолчанию;
- отдельный ограниченный профиль `OPPO_RESCUE_FASTBOOT=yes`;
- watchdog перед передачей управления ядру для диагностики зависаний.

## Fastboot v0.2.0

При включении одиночный или удерживаемый `VOL-` открывает fastboot, двойной
`VOL-` открывает Boot Menu, а `VOL+` выбирает recovery. В fastboot-интерфейсе
`VOL+` переключает действия, `POWER` подтверждает. LK использует физические
коды матрицы/PMIC напрямую и не читает назначение этих кнопок из DTBO.

Экран разблокировки тоже управляется аппаратно: сначала `VOL+` выбирает
`UNLOCK BOOTLOADER`, затем `POWER` подтверждает. Таймера и автоматического
подтверждения нет. Если persistent lock state неожиданно закрыт или не
читается, LK не загружает систему и не перезаписывает `seccfg`, а остается в
fastboot до явного `fastboot flashing unlock`.

Если устройство уже разблокировано, эта команда отвечает `already unlocked`
без ложной надписи `DEVICE IS LOCKED`, повторной записи состояния и очистки
userdata. Опасный выбор выделяется янтарным цветом; красный оставлен для
реальных ошибок.

Полный список новых команд, `getvar`, ограничений `fetch`, проверок
`fastboot boot` и действий экранного меню приведён в
[`FASTBOOT.md`](FASTBOOT.md).
Безопасная прошивка, rescue-профиль и ограничения определения загруженной
копии LK подробно описаны в [`FASTBOOT_SAFETY.md`](FASTBOOT_SAFETY.md).

## Главная найденная причина

Внутри BSP платформа называется `MT6762`, но Android этой прошивки выбирает
свои init/ueventd-файлы по имени `mt6765`. Когда LK записывал в FDT:

```text
/firmware/android/hardware = "mt6762"
```

Android загружал неправильный `ueventd.mt6762.rc`. Из-за этого не выполнялась
штатная цепочка `init.mt6765.rc`, не монтировались `nvcfg`, `data` и другие
разделы, а ошибка выглядела как падение HWC или проблема ядра.

Рабочий вариант сохраняет `MTK_PLATFORM_REVISED=MT6762` для внутренней логики
BSP, но передает Android:

```text
[FDT_ANDROID] hardware=mt6765
```

После этого normal boot использует `ueventd.mt6765.rc`, выполняет `mount_all`
и проходит тот же путь, что стоковый загрузчик.

## Сборка

Нужна Linux/WSL-среда с GNU Make, Perl и Python 2. Из публичной версии
намеренно исключены закрытые signing-ключи и стоковый `main_dtb_header.bin`.
DTB можно хранить только локально, если вы имеете право извлечь его из своей
прошивки. Для unsigned-сборки:

```bash
make -j"$(nproc)" oppo6762_18540 NO_SIGN=yes
```

Ограниченная ремонтная сборка:

```bash
make -j"$(nproc)" oppo6762_18540 NO_SIGN=yes OPPO_RESCUE_FASTBOOT=yes
```

`NO_SIGN=no` допустим только с собственными или разрешенными ключами.
`NO_SIGN=yes` подходит лишь для устройства, чья boot chain принимает unsigned
образ; это не обход secure boot на заблокированном телефоне.

Результат:

```text
build-oppo6762_18540/flash/lk.bin
```

Файл автоматически дополняется до 1 МиБ. Репозиторий не публикует готовые
прошиваемые образы; хеш локального результата зависит от состояния исходников
и метаданных сборки.

## Прошивка

Сначала сохраните резервные копии `lk` и `lk2`. Прошейте и проверьте одну
заведомо рабочую копию, прежде чем заменять вторую. Например, для уже
разблокированного телефона:

```bash
fastboot flash lk2 build-oppo6762_18540/flash/lk.bin
fastboot reboot bootloader
# Проверить сборку и только затем отдельно решать, обновлять ли lk.
```

Наличие `lk2` само по себе не доказывает автоматический fallback. Исследованный
preloader выбирает копию по GPT active bits, но не передает LK достоверный
идентификатор источника, поэтому `active-lk-partition` честно остается
`unknown`.

Этот образ нельзя прошивать на другие модели, ревизии памяти или панели.
Прошивка загрузчика всегда несет риск полного кирпича и потери данных.

Подробности разработки и найденные закономерности находятся в
[OPPO_A1K_PORT.md](OPPO_A1K_PORT.md).
